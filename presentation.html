Section 1

Microservices Mental Model

What is a Microservice?

An independently deployable service organized around a business capability, owning its own data, communicating over the network.

  • Independently deployable — ship without coordinating with other teams
  • Business capability — not a technical layer, a domain boundary
  • Owns its data — no shared database

Monolith vs. Microservices

Monolith

  • Simple to develop and deploy
  • Easy local debugging
  • One database, one codebase
  • No network latency between modules

Microservices

  • Independent scaling
  • Team autonomy
  • Technology flexibility
  • Fault isolation

It's a trade-off, not an upgrade.

Modularity and Distribution

Goal: top-right (microservices). Avoid bottom-right (distributed big ball of mud).

The Cost of Microservices

  • Network is unreliable — calls fail, latency varies
  • Distributed data — no joins across services, eventual consistency
  • Operational complexity — deploy, monitor, debug N services
  • Testing is harder — integration tests span services

Don't use microservices because they're trendy.
Use them when the organizational and scaling benefits outweigh the complexity.

Microservices Patterns

Design

  • Decompose by business capability
  • Decompose by subdomain
  • Strangler Fig / Anti-corruption layer
  • Database per Service ✔

Communication

  • API Gateway / BFF ✔
  • Service Discovery ✔
  • Messaging & Domain Events

Data

  • Saga (distributed transactions)
  • CQRS / Event Sourcing
  • Transactional Outbox

Operations

  • Circuit Breaker ✔
  • Externalized Configuration ✔
  • Distributed Tracing ✔
  • Health Check API ✔

Service Discovery & API Gateway

Spring Cloud Eureka • Gateway Server MVC • LoadBalancer

Service Discovery with Eureka

How it works

  1. Services register with Eureka on startup
  2. Services query Eureka to find other services
  3. Eureka sends heartbeats to detect failures
  4. Client-side load balancing distributes traffic

No more hardcoded URLs

// Before
.baseUrl("http://localhost:8081")

// After
.baseUrl("http://catalog-service")

@Bean
public RouterFunction<ServerResponse> gatewayRoutes() {
    return route("catalog_route")
            .GET("/api/books/**", http())
            .before(uri("lb://catalog-service"))
        .build()
        .and(
            route("order_post_route")
                .POST("/api/orders", http())
                .before(uri("lb://order-service"))
            .build()
        )
        .and(
            route("order_get_route")
                .GET("/api/orders", http())
                .before(uri("lb://order-service"))
            .build()
        );
}

Section 5

Resilience Patterns

Spring Framework Resilience • @Retryable

The Network Will Fail

  • Services go down
  • Network partitions happen
  • Downstream services get slow
  • One failure can cascade through the entire system

The question isn't if things will fail, but how your system behaves when they do.

Built-in Retry in Spring Framework 7

  • Spring Framework 7 includes native retry support
  • No external libraries needed — just @Retryable
  • Enable with @EnableResilientMethods
  • Supports exponential backoff, jitter, max delay

@Configuration
@EnableResilientMethods
public class ResilientConfig {
}  // already provided in the project

Resilience4j

Lightweight, standalone fault-tolerance library (no Spring required)

  • CircuitBreaker — stop calling a failing service
  • Retry — retry failed calls with backoff
  • RateLimiter — limit calls per period
  • TimeLimiter — timeout long-running calls
  • Bulkhead — isolate failures, limit concurrency
  • Cache — cache successful results

Patterns compose via decorators — resilience4j.readme.io

Circuit Breaker Pattern

Prevents cascading failures by short-circuiting calls to a failing service

  CLOSED          OPEN         HALF-OPEN
 ┌──────┐      ┌──────┐      ┌──────────┐
 │calls │      │fails │      │ allow    │
 │pass  │─────►│fast  │─────►│ trial    │
 │thru  │ fail │      │ wait │ calls    │
 └──────┘ rate └──────┘ time └────┬─────┘
     ▲                        ok? │ fail?
     └────────────────────────────┘   │
                                      ▼
                                   OPEN
  • Closed — requests flow normally
  • Open — fail fast, don't call downstream
  • Half-Open — let a few trial requests through

@CircuitBreaker(name = "catalog", fallbackMethod = "fallback")
@Retry(name = "catalog")
public BookResponse getBook(String isbn) {
    return restClient.get().uri("/api/books/{isbn}", isbn)
            .retrieve().body(BookResponse.class);
}

Section 8

When (Not) to Use Microservices

This Bookshop Should Be a Monolith

  • Small team, simple domain
  • 5 services to deploy instead of 1
  • Network calls where function calls would do
  • More infrastructure than business logic

So when should you use microservices?

Consider Microservices When...

  • Multiple teams need to ship independently
  • Parts of the system need to scale differently
  • Different parts have different technology needs
  • You need fault isolation between components
  • Your monolith has become a deployment bottleneck

Start with a monolith. Split when you have a reason to.

The Distributed Monolith Anti-Pattern

  • Services that must be deployed together
  • Shared databases between services
  • Tight coupling via synchronous call chains
  • All the pain of microservices, none of the benefits

If you can't deploy a service independently,
it's not a microservice — it's a distributed monolith.

Anti-Patterns

Integration Database — shared DB couples services. Monolithic Frontend — one UI negates backend independence.

What We Did NOT Cover

  • Distributed transactions & sagas
  • Event-driven communication (Kafka, RabbitMQ)
  • API versioning
  • Security (OAuth2 / JWT)
  • Distributed tracing (Zipkin / Tempo)
  • Kubernetes deployment
  • Contract testing

Each of these deserves its own workshop!
